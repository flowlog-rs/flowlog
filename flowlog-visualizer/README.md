# FlowLog Profile Visualizer

A small CLI that converts **FlowLog profiling outputs** into a single, self-contained HTML report for interactive exploration.

This tool is intended to be used alongside FlowLog's profiler: a profiling build (`flowlog-compiler … -P`) emits, under `<stem>_log/`, a physical-operator-to-logical-node mapping (`ops.json`) and a folder of per-worker unified metrics logs (`metrics/metrics_worker_t{t}_{index}.log`), and this CLI renders them into a browsable visualization.

## What it does

Given FlowLog's profiling artifacts, the CLI:

- Parses **FlowLog's** operator mapping (`ops.json`) and the per-worker unified metrics logs, aggregating across workers (mean + variance).
- Maps operator addresses to logical nodes (inputs, strata, rules operators, runtime operators, inspection).
- Aggregates per-node and per-operator across three dimensions:
  - **Time**: activations and total active time (ms)
  - **Flow**: tuples in / out (`tup_in`, `tup_out`) — exposes intermediate result sizes, e.g. a join's blow-up
  - **Memory**: `batched_in`, merges, `merge_in`, `merge_out`, dropped, and batcher peak bytes/capacity (`bat_bytes`, `bat_cap`)
- Produces a self-contained `report.html` with:
  - A **node list** grouped by block (input / strata / inspect) and, within strata, grouped by **rule vs. runtime**
  - An interactive **DAG view** (pan/zoom + click-to-select) with a **Time / Flow / Memory toggle** to switch color modes
  - A per-node **operator table** with time, flow, and memory columns for detailed drill-down
  - A **top-10 list** showing the hottest nodes by the selected metric
  - A **snapshot selector** for incremental runs — step through each committed transaction's metrics

## Inputs (generated by FlowLog)

This tool consumes files **generated by FlowLog** (a `-P` profiling build, under `<stem>_log/`):

- `ops.json`: operator mapping JSON (logical nodes, blocks/strata, rule stage/runtime tags, operator addresses, DAG edges).
- `metrics/metrics_worker_t{t}_{index}.log`: one unified, fixed-width per-operator table per worker (`t` = committed-transaction index, `0` for batch mode; `index` = worker id), with columns `addr acts active_ms tup_in tup_out arr_in merges merge_in merge_out dropped bat_bytes bat_cap name`. Cells that don't apply to an operator are `n/a`.

> Note: `ops.json` provides the "semantic structure" (nodes and edges); the metrics logs provide the runtime measurements. The report combines them.

## Usage

Run the `flowlog-visualizer` executable against a profiling run's `<stem>_log/`:

```bash
flowlog-visualizer --ops <stem>_log/ops.json --metrics <stem>_log/metrics --out report.html
```

Then open `report.html` in your browser.

### CLI flags

| Flag | Short | Description |
|------|-------|-------------|
| `--ops <PATH>` | `-p` | Path to FlowLog operator mapping (`ops.json`). |
| `--metrics <PATH>` | `-m` | Path to the folder of per-worker metrics logs (`<stem>_log/metrics/`). |
| `--out <PATH>` | `-o` | Output HTML file to write (e.g., `report.html`). |

All three flags are required.

## Output

- `report.html`: a standalone HTML file (no external assets) that you can share or archive alongside the profiling run.

## Typical workflow

1. Compile and run your FlowLog program with profiling enabled (`flowlog-compiler … -P`, then run the binary) to produce `<stem>_log/ops.json` and `<stem>_log/metrics/`.
2. Run `flowlog-visualizer` to generate `report.html`.
3. Use the report to identify hot nodes/operators and understand how work is distributed across components -- switch between **Time**, **Flow**, and **Memory** views using the toggle in the report.
